	"Yes, but only in a very limited sense, and not in the way most people mean when they talk about macOS “system daemons” or “agents.”

### What actually runs before FileVault unlock

When FileVault is enabled, the machine boots into a small pre‑boot environment that lives on an unencrypted volume and is separate from the main macOS system. This environment’s job is basically just to present the unlock UI and obtain a valid credential to decrypt the main APFS Data volume. [community.jamf](https://community.jamf.com/general-discussions-2/auto-login-with-filevault-enabled-29417)

In that state:

- The encrypted system volume is not mounted, so anything stored there (including most launchd plists, binaries, scripts, and user home data) is inaccessible. [help.swif](https://help.swif.ai/en/articles/10705969-troubleshooting-a-non-admin-login-failure-on-filevault-enabled-macos)
- What runs is Apple’s own pre‑boot code and components, not the normal launchd hierarchy from the full OS. [community.jamf](https://community.jamf.com/general-discussions-2/auto-login-with-filevault-enabled-29417)

So you cannot have an arbitrary launch daemon or launch agent (in `/System/Library/LaunchDaemons`, `/Library/LaunchDaemons`, `/Library/LaunchAgents`, or `~/Library/LaunchAgents`) start before FileVault unlock, because the volume those items live on is still encrypted and not yet available. [stackoverflow](https://stackoverflow.com/questions/38828858/os-x-launch-daemon-doesnt-start-until-user-logs-in)

### When normal LaunchDaemons and LaunchAgents start

Once a valid FileVault user unlocks the disk, macOS continues booting into the full OS, mounts the data volume, and only then does the normal `launchd` world come up:

- System **LaunchDaemons** (in `/System/Library/LaunchDaemons` and `/Library/LaunchDaemons`) start after the data volume is unlocked but do not depend on a user GUI login session. [news.ycombinator](https://news.ycombinator.com/item?id=45296353)
- **LaunchAgents** (per-user or in `/Library/LaunchAgents`) require a user login session and thus start later, after login. [discussions.apple](https://discussions.apple.com/thread/254467386)

From the point of view of a FileVault‑enabled machine that just powered on unexpectedly, any custom daemon you define will not run until someone enters a password at the FileVault screen and the OS proper boots. [community.jamf](https://community.jamf.com/general-discussions-2/auto-login-with-filevault-enabled-29417)

### Practical implication

If your requirement is:

- “Run this code even after a power loss **without** any human entering a password”  
then you cannot have full‑disk FileVault enabled, because that scenario inherently blocks unattended boot until a user unlocks the volume. [reddit](https://www.reddit.com/r/macsysadmin/comments/1achkcz/allow_systemlevel_launchd_processes_while/)

If your requirement is:

- “I’m okay manually entering the FileVault password after reboots, but I want services to be running even when no one is logged in”  

then you should:

- Enable FileVault.  
- Install your services as **LaunchDaemons** in `/Library/LaunchDaemons`.  
- They will start automatically once the disk is unlocked and boot completes, even if no user logs in to the GUI. [news.ycombinator](https://news.ycombinator.com/item?id=45296353)

There is no supported way for third‑party code to run from the encrypted system volume **before** that FileVault unlock step."
